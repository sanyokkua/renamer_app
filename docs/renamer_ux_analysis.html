Section 02

Overall Layout Architecture

The app uses a three-zone split layout: a left configuration panel, a central file table, and a bottom metadata strip. This is a reasonable structural decision for a utility app — but the execution has significant issues.

✓ Logical Three-Zone Structure Good

Separating "how to rename" (left) from "what will be renamed" (center/right) is fundamentally sound. This mirrors the mental model of the task: first configure, then review, then act. Power users who work with this app daily will appreciate the persistent side panel that stays visible without needing to open dialogs.

✗ Wasted Vertical Space on Left Panel Bad

In Image 1, the left panel has significant empty space below the "Text to add" field. In Image 2, the datetime mode uses the space better but still leaves half the panel empty. This creates an unbalanced composition. JavaFX's VBox with a Spacer or a ScrollPane would allow the space to be used, for example for live format previews, tips, or a mini changelog of recent operations.

✗ No Drag-to-Resize Between Panels Bad

The left panel appears to have a fixed width, and there is no visible SplitPane divider between the configuration panel and the file table. Users with many files benefit from collapsing the config panel to maximize the table, or expanding it to see long format strings. JavaFX's SplitPane natively solves this with zero custom code.

✗ No Collapsible Left Panel or Compact Mode Bad

Once the user has configured the renaming mode, the left panel is purely visual clutter during the review phase. There should be a collapse toggle (arrow icon on the divider) to temporarily hide it. This is a common productivity app pattern (VS Code, Finder's sidebar, etc.) and JavaFX supports it cleanly with a SplitPane and animated width transition.

Section 03

Color System & Visual Feedback

Color is the most immediately striking aspect of this application — and it is the most problematic. The current choices create visual strain and undermine the app's credibility as a professional tool.

✗ Aggressive Full-Row Cyan Selection Color Critical

Every row in the file table is painted with a vivid #00BCD4-style cyan fill. This is not a selection state — it appears to be the default row style. This has multiple severe consequences:

  • Eye fatigue: High-saturation cyan on large table areas causes rapid visual fatigue, especially in a utility where users stare at file lists for extended periods.
  • Loss of selection signal: If every row looks "selected," the user cannot tell what is actually selected. The visual affordance of selection is destroyed.
  • Loss of status differentiation: When everything is the same cyan, there is no remaining color space for meaningful status states without clashing horribly.
  • Professionalism damage: This color palette is reminiscent of JavaFX 2 default themes from 2012. It immediately signals to users that the app is outdated.
🔴

Recommendation: Remove the cyan row background entirely. Use neutral alternating row backgrounds (white / #f8f8f8). Reserve color exclusively for status states: subtle gray for "not processed," blue-tinted for "previewed," green for "success," red for "error."

✓ Green Success Rows Are Semantically Correct Good

In Image 3, after renaming, all rows turn green to signal success. This is the right instinct — using color to convey operation status. The semantic choice (green = success) is universally understood. This pattern should be preserved and extended to a proper status system.

✗ No Neutral / Default State Differentiation Bad

The current palette only shows two states: cyan (default/loaded) and green (renamed). A real-world batch renaming workflow needs at minimum five visual states: Default (unprocessed), Previewed (will rename), Conflict (name already exists), Warning (skipped), Success (renamed), and Error (failed). The current two-tone system cannot express these without adding a sixth state that visually clashes.

Section 04

Typography & Readability

The application appears to use the system default font with no custom sizing hierarchy. This is functional but loses the opportunity to create clear information hierarchy.

✗ No Visual Hierarchy in Text Bad

All labels in the left panel (field names, radio button labels, dropdown labels) appear at identical size and weight. Without typographic variation, the eye cannot quickly navigate the configuration options. A user switching between modes must re-read everything to understand context. Applying font-weight: 600 to field labels and keeping values at regular weight creates instant scanability.

✗ Filename Truncation is Invisible Bad

Long filenames or paths will be silently clipped by column widths. There is no indication of truncation (no ellipsis, no tooltip-on-hover defined). In a file renaming app, the exact filename is the most critical piece of data. Truncating it silently is a significant UX failure. JavaFX TableColumn supports setResizable(true) and cell tooltips via Tooltip on cell factory — these should be enabled for every column.

→ Typography Improvements Improve

  • Use 13–14px for table content (current appears fine), 12px for metadata labels, 11px for secondary metadata values.
  • Apply font-weight: 700 to section labels in the left panel ("Add text to:", "Date format:") so they read as field labels, not content.
  • The "Changed Name" column should use a monospace font (e.g., system monospace) so users can easily spot character-level differences from the original filename.
  • The "File Status" text ("No any actions occurred") has a grammatical error. It should read "No actions performed" or "Awaiting rename." Fix copy during the redesign pass.
  • Use CSS styling via JavaFX CSS to define font scales rather than inline sizes, keeping them consistent across modes.
💡

JavaFX Tip: Apply a custom stylesheet with scene.getStylesheets().add(...) and define .table-cell, .label-field, and .metadata-value CSS classes for consistent typography control across the entire app.

Section 05

Mode Configuration Panel (Left)

The configuration panel on the left side is the primary interaction area. Its design directly affects how quickly users can configure and execute renames.

✓ Mode Dropdown Is Logically Placed at Top Good

Placing the mode selector as the top element is correct. It's the primary choice that changes everything below it. This follows a top-down cognitive flow: "What mode am I in?" → "Configure it" → "See the result." The top position ensures it's the first thing users interact with.

✗ No Visual Separation Between Mode Selector and Mode Options Bad

In Image 2, the "Renaming mode:" label appears again directly below the dropdown — which is also labeled "Renaming mode:" in the dropdown itself. This creates immediate confusion: is this a sub-mode, a related setting, or just repetition? The left panel radio group ("Begin / End / Replace") is actually a property of the datetime mode, but it's presented ambiguously under a repeated "Renaming mode:" label. A clear visual separator (a horizontal rule or a section header with a different background) between the global mode selector and its specific options is essential.

✗ No Live Preview of the Format in the Panel Bad

When configuring "Date format: YYYYMMDD" and "Time format: HHMMSS", the user must mentally calculate what the output will look like. There is no example output shown in the panel. A simple static example like Example: photo.jpg → 20240411_083653_photo.jpg displayed as a live-updating preview would eliminate confusion and dramatically reduce configuration errors. This is a single Label bound to the form fields — minimal implementation effort, massive UX gain.

✗ Options Feel Crammed Without Breathing Room Bad

The spacing between elements in the left panel is inconsistent and generally too tight, particularly in the datetime mode (Image 2) where 7 different fields are stacked vertically with minimal padding between them. This makes scanning the options feel overwhelming. Each logical group (position, date options, time options, source options) should have visible separation — either via a light separator line, a subtle background tint, or simply more vertical padding.

Section 06

File List & Table Design

The file table is the centerpiece of the application. It is where users spend most of their time during review. Every detail matters here.

✓ Four-Column Structure Is Logical Good

Name → Type → Changed Name → File Status is a correct and logical column progression. It maps directly to the "before → category → after → outcome" mental model. This should be preserved.

✗ "Type" Column Has Low Value, Wastes Space Bad

The "Type" column shows only "File" for every row in all three screenshots. If the app only handles files (not directories), this column is completely useless — it provides zero information. If it can show "Directory" or other types, it should still be a very narrow column with an icon instead of text ("📄" vs "📁"). Text-based "File" repeated 16 times occupies significant horizontal space and adds zero value. Replace with a small file-type icon column (HEIC → image icon, JPG → image icon) that is immediately recognizable and takes 40px instead of 80px.

✗ No Row Numbering or File Count Context Bad

The user loads 16+ files but has no quick way to know how many total files are in the list without scrolling through. A row number column (#) and a status bar showing "16 files loaded, 0 renamed" provides essential orientation context, especially for large batches of hundreds of files.

✗ No Sortable Columns Bad

JavaFX's TableView supports column sorting natively via column.setSortable(true). If this is not enabled, users cannot sort files alphabetically, by type, by changed name, or by status — all highly useful operations when reviewing a large batch. When renaming 200 photos, sorting by status to see all errors at the top is a critical workflow step.

✗ No Filtering or Search Bad

For batches of hundreds of files, there is no way to filter the list (e.g., "show only errors," "show only .HEIC files," "search for filename"). JavaFX's FilteredList wrapping an ObservableList makes this trivial to implement. A search bar above the table that filters by name would be a high-impact addition.

✗ No Visual Diff Between Original and Changed Name Bad

When the original filename is "20240411_083653.HEIC" and the changed name is "Text_20240411_083653.HEIC", the only way to spot the difference is to read both carefully. For longer filenames or replacement operations, this becomes very error-prone. The "Changed Name" cell should visually highlight the changed portion — for example, showing the added prefix in a colored/bold span. This requires a custom cell factory with a TextFlow inside the cell, which is fully supported in JavaFX.

Section 07

Metadata Panel (Bottom)

The bottom metadata strip (visible in Image 1) shows per-file information for the selected row. This is a genuinely useful feature, but its current implementation misses several opportunities.

✓ Showing Per-File Metadata is Excellent Good

Displaying Absolute Path, Name, Extension, Type, MIME, File Creation Date, Modification Date, Content Creation Date, Width, and Height in a panel below the selection is a genuinely strong feature. This information is essential when troubleshooting why a file was renamed a certain way (e.g., "why did it use this date? — oh, the content creation date was different from the file creation date"). No competing batch renaming apps show this level of metadata inline. This should be a selling point and should be designed to look excellent, not like an afterthought.

✗ Metadata Disappears When datetime Mode Is Selected Bad

In Images 2 and 3, the metadata panel is completely absent. This is likely because the left panel's configuration options take up the left-bottom space. But the metadata panel was in the right-center area in Image 1, not the left. Its disappearance in the datetime mode appears to be a layout bug or an inconsistency. The metadata panel should always be present when a file is selected, regardless of which mode is active.

✗ Two-Column Key-Value Layout Wastes Space and Looks Primitive Bad

The current metadata display is a simple two-column label list ("Absolute Path" → long value). This wastes horizontal space (the label column is very wide while values could span further) and has no visual grouping. A more sophisticated design would group fields into categories (Identity, Dates, Media Properties), use a more compact grid, and distinguish label text from value text clearly through weight and color.

✗ No Resizable / Collapsible Metadata Panel Bad

The metadata panel appears to take up approximately 35% of the total application height when visible. This is excessive when reviewing a large list of files. The panel should be resizable (SplitPane horizontally with the table) and collapsible via a toggle, so users can maximize the file table when they don't need metadata and expand it when they do.

Section 08

Controls, Buttons & Actions

The action buttons at the bottom of the screen control the core renaming workflow. Their design has several issues that create unnecessary risk and confusion.

✗ Truncated Button Labels Are Unacceptable Critical

The buttons "Auto Pr…", "Preview r…", and "Rename f…" have their labels truncated. This means the user cannot read what these buttons do. In a utility where one button renames files permanently, truncating the "Rename files" button is a serious UX failure. It creates genuine risk: a new user might click "Rename f…" without understanding it's an irreversible action. Fix: either make the buttons wider, use shorter labels ("Auto Preview", "Preview", "Rename"), or use icon+tooltip buttons.

✗ No Visual Hierarchy Between Primary and Secondary Actions Bad

All four buttons ("Auto Pr…", "Preview r…", "Rename f…", "Clear") appear at the same visual weight. The rename action is the primary action — it should be a filled primary-color button. The preview action is secondary — outlined. Clear is destructive — it should be a text button or have warning styling. In JavaFX, this is achieved purely through CSS classes: .button-primary, .button-secondary, .button-danger.

✗ No Confirmation Dialog Before Renaming Bad

Renaming 16+ files is a significant irreversible operation. The app appears to execute immediately on clicking "Rename f…" without any confirmation. A brief confirmation dialog — or even better, a progress indicator with a Cancel button — is essential for a batch operation tool. JavaFX's Alert with AlertType.CONFIRMATION handles this in five lines of code.

✗ "Auto Preview" Checkbox is Ambiguous Bad

The checkbox labeled "Auto Pr…" (likely "Auto Preview") in the bottom-left suggests that preview is computed automatically as settings change. But it's not clear: does this mean the preview updates in real-time? Does it run on every keystroke? Does it disable the "Preview r…" button? The checkbox placement (bottom-left, next to action buttons) is unusual — checkbox-type settings belong in the configuration panel, not next to action buttons. Move it to the left panel under the mode options.

✓ "Reload File" Appears After Renaming Good

In Image 3, after renaming, a "Reload File" button appears. This is a good contextual action — it appears exactly when needed (after a rename operation, to refresh the view). This pattern of contextual button appearance should be applied more broadly.

Section 09

Analysis of Application Modes

The app has multiple renaming modes. Only two are visible in the screenshots, but we can analyze the structural patterns these modes reveal, and what any mode configuration screen should do well.

✓ Dynamic Left Panel Per Mode

The left panel changes its content based on the selected mode. This is the correct approach — don't show all options at once, show only what's relevant. The implementation is sound; the visual design needs improvement.

✗ Mode Switching Has No Transition

When switching modes, the left panel content changes instantly with no visual transition. A subtle fade or slide animation (150ms) would make the context switch feel polished and intentional, not jarring.

✗ Likely Modes Not Yet Analyzed — Structural Risk Important

Based on the visible screens, the app likely has modes such as: "Sequential numbering," "Replace text in name," "Change case," "Remove characters," "Insert at position," and possibly more. Each of these modes faces the same structural risk: if the left panel design isn't systematic, each mode will look slightly different from the others, creating inconsistency. A proper mode system should define a template for every mode panel:

  • Zone A: Mode header (name + one-line description)
  • Zone B: Position option (Begin / End / Replace) — if applicable to the mode
  • Zone C: Mode-specific options (grouped, labeled clearly)
  • Zone D: Live example preview (always visible)

This template ensures every mode looks and behaves consistently even if its specific fields differ.

Section 10

UX Flows & Interaction Patterns

Beyond individual components, the overall interaction flow — how a user goes from opening the app to successfully renaming files — has systemic gaps.

✗ No Visible Drop Target for File Import Bad

Looking at all three screenshots, there is no obvious way to add files to the list. There's no "Add Files" button, no "Open Folder" button, no drag-and-drop zone visible. The files appear already loaded. For a new user, the very first step — "how do I load files?" — is completely opaque. A prominent drop zone (either a top toolbar area or an empty-state illustration when the table is empty) is essential for discoverability.

✗ Empty State is Presumably Invisible Bad

When no files are loaded, the table is presumably just empty rows or completely blank. JavaFX's TableView.setPlaceholder(Node) allows setting a custom empty state: an illustration, "Drop files here to begin" message, and an "Open Files…" button. This is the most powerful onboarding moment and should not be wasted on a blank table.

✗ No Undo Capability Critical

Batch renaming is an inherently risky operation. After executing a rename, there is no visible way to undo the changes (restore original filenames). The "Reload File" button reloads the view but doesn't reverse the rename. An "Undo Last Rename" button that restores all files to their pre-rename names (by keeping a rename log in memory) is a critical safety feature. Many professional renaming tools (A-Rename, Bulk Rename Utility, Retcon) offer this. JavaFX's UndoManager pattern or a simple reverse-rename list is the implementation path.

✗ No Keyboard Shortcuts Visible Bad

Power users who do batch renaming regularly will heavily rely on keyboard navigation. There are no visible keyboard shortcuts for Preview (⌘R?), Rename (⌘↵?), or Clear (⌘⌫?). JavaFX's KeyCombination and Accelerator on MenuItem support this. At minimum, the primary destructive action (Rename) should be guarded with a key combination that prevents accidental trigger (not just Enter — something like ⌘⇧R).

Section 11

Modernization Roadmap (JavaFX-Native)

Every recommendation in this document can be implemented entirely within JavaFX without third-party UI libraries. Here is a concrete technical roadmap.

→ CSS Theming — Replace Everything Visual

Create a styles.css applied to the root Scene. Define:

  • .root: Background #F8F7F4, font-family system-ui or a bundled font
  • .table-row-cell: Remove default alternating colors, set transparent background, add hover state (#F0EDE6), add selected state (#E8F0FF with border)
  • .table-row-cell.status-success: Background #F0FAF4
  • .table-row-cell.status-error: Background #FEF2F2
  • .button-primary: Background accent color, white text, no border, 8px border-radius
  • .button-secondary: Transparent background, 1px border, accent color text
  • .split-pane-divider: 1px solid #DDD9D0, no default JavaFX arrow

→ TableView Enhancements

  • Use custom cell factories with TextFlow for diff-highlighted "Changed Name" column
  • Use GraphicTableCell for the Type column (icon instead of text)
  • Wrap the ObservableList in FilteredList and SortedList for free filtering+sorting
  • Add a TableColumn for row number using a custom cell factory with getIndex()
  • Set setPlaceholder() with a proper empty-state node
  • Enable column reordering via setReorderable(true)
  • Persist column widths to user preferences (Preferences.userNodeForPackage())

→ Layout & Panels

  • Wrap left panel and main table in a SplitPane with horizontal orientation
  • Add a collapse toggle button on the divider (programmatically hide/show left panel)
  • Wrap table and metadata panel in a SplitPane with vertical orientation
  • Use TitledPane for collapsible option groups within the left panel
  • Use VBox with Separator nodes to group sections visually

→ Transitions & Polish

  • Use FadeTransition (150ms) when switching mode panels in the left side
  • Use Task<Void> with a ProgressBar popup during file renaming operations
  • Use Tooltip on all truncated cells and abbreviated buttons
  • Use Alert with AlertType.CONFIRMATION before large batch renames
  • Bind button disabled state to file list size (JavaFX property binding)
  • Support drag-and-drop file loading via setOnDragOver / setOnDragDropped on the TableView

Closing

Summary

This application has solid functional bones. The renaming capabilities, metadata display, and multi-mode architecture demonstrate real engineering investment. However, the visual design is stuck a decade in the past, and several UX patterns actively work against the user's trust and efficiency.

✓ What Is Genuinely Good

  • The three-zone layout concept (configure → review → act) is architecturally sound
  • The per-file metadata panel is a differentiating feature competitors lack
  • Multi-mode architecture with dynamic left panel is the right approach
  • The semantic shift to green after successful rename is correct
  • Contextual "Reload File" button after rename shows good reactive thinking
  • The datetime mode with format customization is powerful and flexible

✗ What Must Change

  • The cyan table row background — the single most damaging visual decision
  • Truncated button labels — creates genuine ambiguity around a destructive action
  • Absence of Undo — unacceptable for a batch file operation tool
  • No live preview in the configuration panel — forces users to guess
  • No empty state or drag-and-drop — first-time user experience is opaque
  • No confirmation before rename — puts user data at risk
  • Duplicate "Renaming mode:" labels — actively confuses the interface hierarchy
🎯

The Core Principle: A batch file renaming tool handles irreplaceable files. Every design decision should be measured against the question: "Does this help the user feel confident and in control before they commit to renaming?" The current design does not inspire confidence. The recommended changes, particularly live preview, proper status feedback, undo capability, and visual calm (removing the cyan), would collectively transform this from a technically functional tool into one that feels genuinely trustworthy and professional.
